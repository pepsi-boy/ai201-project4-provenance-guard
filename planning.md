# Provenance Guard — Planning & Spec

> Written before implementation (Milestone 2). This document is the design contract that
> every module implements against, and the primary context handed to AI tools during
> Milestones 3–5.

## The Problem

Creative platforms need to tell readers whether a submitted piece of writing was made by a
human or generated by AI — not to police creativity, but to protect attribution and give
readers honest context. Perfect AI detection is an unsolved problem, so the real engineering
task is to **classify with two independent signals, communicate uncertainty honestly, and
give miscategorized creators a path to appeal.**

A guiding constraint runs through every decision below:

> **A false positive (labeling a human's work as AI) is worse than a false negative.**
> The system is deliberately biased toward "uncertain" and "human" over "AI" in the
> borderline zone, and every "likely AI" verdict ships with a visible appeal path.

## Architecture

### Submission flow

```
                        POST /submit
                    { text, creator_id }
                            │
                            ▼
                 ┌──────────────────────┐
                 │  input guard          │  reject empty / too-short text
                 │  (len, word count)    │  (<40 words → forced "uncertain")
                 └──────────┬───────────┘
                            │ raw text
              ┌─────────────┴─────────────┐
              ▼                            ▼
   ┌────────────────────┐      ┌──────────────────────┐
   │ SIGNAL 1: LLM       │      │ SIGNAL 2: Stylometry  │
   │ (Groq llama-3.3)    │      │ (pure Python)         │
   │ semantic / holistic │      │ structural / stats    │
   └─────────┬──────────┘      └───────────┬──────────┘
             │ llm_score 0–1                │ stylometric_score 0–1
             │ (P that text is AI)          │ (P that text is AI)
             └──────────────┬───────────────┘
                            ▼
              ┌──────────────────────────────┐
              │  CONFIDENCE SCORING            │
              │  combined = 0.65·llm + 0.35·sty│
              │  confidence = f(decisiveness,  │
              │                 agreement)     │
              │  → attribution band            │
              └──────────────┬────────────────┘
           combined_score, confidence, attribution
                            ▼
              ┌──────────────────────────────┐
              │  TRANSPARENCY LABEL            │  maps band → plain-language text
              │  (3 variants)                  │
              └──────────────┬────────────────┘
                             │ label text
              ┌──────────────┴────────────────┐
              ▼                                ▼
   ┌────────────────────┐          ┌──────────────────────┐
   │ AUDIT LOG (SQLite)  │          │  JSON RESPONSE        │
   │ append event        │          │  content_id, result,  │
   │ + submissions row   │          │  confidence, label    │
   └────────────────────┘          └──────────────────────┘
```

### Appeal flow

```
   POST /appeal { content_id, creator_reasoning }
                     │
                     ▼
       ┌───────────────────────────┐
       │ look up submission by id   │  404 if unknown content_id
       └─────────────┬─────────────┘
                     │ original decision
                     ▼
       ┌───────────────────────────┐
       │ status → "under_review"    │  store creator_reasoning
       │ (submissions table)        │
       └─────────────┬─────────────┘
                     │
                     ▼
       ┌───────────────────────────┐
       │ AUDIT LOG: append "appeal" │  logged alongside original classification
       │ event referencing content  │
       └─────────────┬─────────────┘
                     │
                     ▼
            JSON confirmation { status: under_review }
```

**Narrative.** On `/submit`, raw text is guarded for length, then scored by two independent
signals — a Groq LLM (semantic judgment) and a pure-Python stylometric analyzer (structural
statistics). Their two `P(AI)` scores are combined into a single `ai_probability` and a
`confidence` value that folds in both how decisive the score is and how much the two signals
agree; that maps to one of three attribution bands, which selects one of three transparency
labels. Every submission writes a structured row to the audit log and a current-state row to
the submissions table, and the label + scores are returned as JSON. On `/appeal`, a creator
submits their `content_id` and reasoning; the system flips that submission's status to
`under_review`, stores the reasoning, and appends an `appeal` event to the same audit log so a
human reviewer can see the appeal next to the original decision.

## The Five Spec Questions

### 1. Detection signals

Two genuinely independent signals — one **semantic**, one **structural**.

| # | Signal | What it measures | Output |
|---|--------|------------------|--------|
| 1 | **LLM classification (Groq `llama-3.3-70b-versatile`)** | Holistic read of whether the text *reads* as AI-generated — coherence patterns, over-smoothness, hedging, generic framing, "AI voice." | `llm_score` ∈ [0,1] = P(AI), plus a short rationale string. |
| 2 | **Stylometric heuristics (pure Python)** | Measurable statistical structure of the prose. | `stylometric_score` ∈ [0,1] = P(AI). |

**Stylometric sub-metrics** (each normalized to a 0–1 "AI-likeness" contribution, then weighted):

- **Sentence-length burstiness** — `stdev(sentence_len)/mean(sentence_len)`. Humans vary
  sentence length a lot (bursty); AI trends uniform. *Low burstiness → more AI-like.*
- **Type-token ratio (vocabulary diversity)** — `unique_words / total_words`. AI reuses a
  narrower vocabulary in a given passage. *Lower TTR → more AI-like.*
- **Contraction & informality rate** — contractions (`don't`, `i'm`), lone lowercase `i`,
  casual markers. Humans use far more in casual registers. *Low rate → more AI-like.*
- **AI-marker phrase density** — connective/boilerplate phrases AI overuses ("it is important
  to note", "furthermore", "moreover", "in conclusion", "delve", "tapestry", "navigate the
  landscape"). *High density → more AI-like.*
- **Punctuation variety** — humans use dashes, ellipses, parentheses, question/exclamation
  marks more freely. *Low variety → more AI-like.*

**Why these two together:** one is semantic (does it *mean* like a human wrote it?), one is
structural (does it *count* like human writing?). They fail differently, which is the whole
point — see confidence scoring and edge cases.

**Blind spots (every signal has them):**

- **LLM signal** can't be trusted on very short text, is itself biased against non-native and
  highly formal human writers (the false-positive risk), can hallucinate a confident rationale,
  and is non-deterministic. It also has no ground truth — it's pattern-matching "AI voice,"
  which humans can now imitate and AI can be prompted to avoid.
- **Stylometric signal** is fooled by any human writing that happens to be uniform and formal
  (academic prose, legal text, non-native writers), by short text (variance is undefined), and
  by poetry/lists/code where the statistics don't mean what they mean in prose. It also can't
  read *meaning* at all — a factually incoherent but statistically "human" text sails through.

### 2. Uncertainty representation

Both signals emit `P(AI)` in [0,1]. Combine:

```
combined_ai_probability = 0.65 * llm_score + 0.35 * stylometric_score
```

The LLM is weighted higher because holistic judgment outperforms surface statistics on most
prose, but stylometry gets a real 35% vote so a confident-but-wrong LLM can be pulled back
toward uncertain.

**Confidence is not the same as `ai_probability`.** `confidence` answers "how sure are we of
*whatever* verdict we're giving?" It is high only when the score is decisive **and** the two
signals agree:

```
decisiveness = 2 * |combined - 0.5|          # 0 at the fence (0.5), 1 at the extremes
agreement    = 1 - |llm_score - stylometric_score|   # 1 = signals agree, 0 = opposite
confidence   = round(0.6 * decisiveness + 0.4 * agreement, 2)
```

So a `combined = 0.9` where the two signals *disagree* (LLM 1.0, stylometry 0.4) yields lower
confidence than a `combined = 0.9` where they agree — which is exactly the honesty we want.

**Bands → attribution (asymmetric, biased against false positives):**

| Condition | Attribution | Rationale |
|-----------|-------------|-----------|
| `ai_probability ≥ 0.70` | `likely_ai` | **High bar** to accuse a human. |
| `ai_probability ≤ 0.40` | `likely_human` | Lower bar — the safe direction. |
| `0.40 < ai_probability < 0.70` | `uncertain` | The honest default for borderline text. |

The uncertain band is **wider on the AI side** (0.40→0.70) on purpose: content leans "AI"
only when both signals really push there. Text under 40 words is force-routed to `uncertain`
with capped confidence regardless of score, because neither signal is reliable at that length.

**What 0.5 means to a user:** roughly "we genuinely can't tell." That's why 0.5 lands squarely
in the uncertain band and produces the uncertain label — not a coin-flip verdict.

### 3. Transparency label design

Three variants, plain language, honest about being an *estimate*. `{c}` = confidence as a
percentage.

**High-confidence AI (`likely_ai`):**
> 🤖 **Likely AI-generated.** Our analysis suggests this piece was probably produced with an
> AI writing tool (confidence: {c}%). This is an automated estimate, not a certainty — detection
> is imperfect. If you created this yourself, you can appeal this label and a human will review it.

**High-confidence human (`likely_human`):**
> ✍️ **Likely human-written.** Our analysis found no strong signs of AI generation in this piece
> (confidence: {c}%). This is an automated estimate and not a guarantee of authorship.

**Uncertain (`uncertain`):**
> ❓ **Attribution uncertain.** We couldn't confidently tell whether a person or an AI tool wrote
> this piece (confidence: {c}%). We'd rather say so honestly than guess — please don't read this
> as a verdict either way.

Only the `likely_ai` label mentions appeals prominently, because that's the label that can
harm a creator. (Appeals are technically allowed on any classification.)

### 4. Appeals workflow

- **Who:** any creator, by submitting the `content_id` from their `/submit` response plus free-text
  reasoning. (No auth in this prototype — noted as a real-system gap.)
- **What they provide:** `content_id` + `creator_reasoning`.
- **What the system does:** looks up the submission (404 if unknown), sets its status to
  `under_review`, stores the reasoning, and appends an `appeal` event to the audit log
  referencing the original classification. **No automated re-classification** (per spec).
- **What a reviewer sees:** `GET /appeals` returns the queue — each item shows the original
  attribution, both signal scores, confidence, the creator's reasoning, and timestamps, so a
  human can judge the original decision against the creator's claim.

### 5. Anticipated edge cases

1. **Repetitive, simple-vocabulary poetry** — low burstiness + low TTR make stylometry scream
   "AI," but it's a stylistic human choice. The LLM signal *may* recognize the poetic form; when
   the two disagree, confidence drops and we fall to `uncertain` rather than `likely_ai`. Still a
   known weak spot.
2. **Non-native / highly formal human writer** — uniform sentence length, formal connectives,
   few contractions push *both* signals toward AI. This is the canonical false positive and the
   reason the AI band starts at 0.70 and every AI label carries an appeal path. (Our test set
   includes exactly this case — the "monetary policy" paragraph.)
3. **Very short text (<40 words)** — variance undefined, LLM unreliable. Force `uncertain`.
4. **Lightly human-edited AI text** — genuinely mid-range; landing in `uncertain` is *correct*,
   not a failure.
5. **Non-prose (code, lists, tables)** — stylometric assumptions don't hold; documented as
   out-of-scope for reliable scoring.

## AI Tool Plan

How I'll use an AI coding tool across the three build milestones, and how I'll verify its output.

### M3 — submission endpoint + first signal
- **Spec I'll provide:** the *Detection signals* section (signal 1) + the *Architecture* diagram.
- **Ask for:** Flask app skeleton with a `POST /submit` stub and the Groq `llm_signal(text)`
  function returning `{ai_probability, rationale}`.
- **Verify:** call `llm_signal` directly on 3 known inputs and eyeball the scores/rationales
  before wiring into the route; confirm the route returns the JSON shape in my contract
  (`content_id`, `attribution`, `confidence`, `label`). Edit signatures to match this spec before use.

### M4 — second signal + confidence scoring
- **Spec I'll provide:** *Detection signals* (signal 2 sub-metrics) + *Uncertainty representation*
  + the diagram.
- **Ask for:** `stylometric_signal(text)` computing the listed sub-metrics, and
  `score(llm_score, stylometric_score)` implementing the exact weighting, confidence formula,
  and band thresholds above.
- **Verify:** run the 4 provided calibration inputs (clear-AI, clear-human, formal-human,
  edited-AI); confirm scores span the range and map to the right bands. **Check the generated
  thresholds against this doc** — AI tools often invent plausible-but-wrong cutoffs. Correct any
  drift before wiring in.

### M5 — production layer
- **Spec I'll provide:** *Transparency label design* (3 variants verbatim) + *Appeals workflow*
  + the diagram.
- **Ask for:** `build_label(attribution, confidence)` returning the exact text per variant, and
  the `POST /appeal` endpoint (+ Flask-Limiter config, audit-log writes).
- **Verify:** assert all three label variants are reachable and the text matches this doc
  character-for-character; submit → appeal → `GET /log` and confirm status flips to
  `under_review` and the appeal event is logged; fire 12 rapid requests and confirm 429s appear
  after the limit.
